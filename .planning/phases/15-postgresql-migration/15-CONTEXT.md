# Phase 15: PostgreSQL Migration - Context

**Gathered:** 2026-03-01
**Status:** Ready for planning

<domain>
## Phase Boundary

Swap the database engine from SQLite (`better-sqlite3`) to PostgreSQL, updating the Drizzle ORM adapter, connection client, migration runner, and environment config. All existing functionality stays identical — same API, same schema shape, same 3 tables. Multi-floor schema changes are Phase 16.

</domain>

<decisions>
## Implementation Decisions

### PostgreSQL provider
- Target: **Neon** (free tier, cloud-hosted, aligns with DEPL-03 in REQUIREMENTS.md)
- Connection via standard PostgreSQL connection string (`DATABASE_URL`)
- No serverless/edge HTTP driver needed — server runs on `@hono/node-server` (Node.js)

### Local development database
- **Docker PostgreSQL** — full parity with production, no local/prod divergence risk
- `docker-compose.yml` with a `postgres` service, exposing `DATABASE_URL` for local dev
- `.env` holds `DATABASE_URL`; `.env.example` documents the expected format

### Data migration approach
- **Fresh seed from `campus-graph.json`** — PostgreSQL starts empty; `seedIfEmpty()` reimports from the JSON asset at startup
- No SQLite-to-PostgreSQL data porting needed — JSON file is the source of truth
- Existing `seedIfEmpty()` function is rewritten for the new driver but logic stays the same

### Drizzle adapter & driver
- Replace `drizzle-orm/better-sqlite3` → **`drizzle-orm/postgres-js`**
- Replace `better-sqlite3` package → **`postgres` (postgres.js)** package
- Remove `@types/better-sqlite3` from devDependencies
- Drizzle schema types change from sqlite-core → **pg-core** (e.g., `pgTable`, `integer`, `text`, `real`, `boolean`)
- Migration runner: `drizzle-orm/postgres-js/migrator` (same API, different import)

### Schema structure
- Tables stay identical: `nodes`, `edges`, `graphMetadata`
- Same columns and constraints — only the Drizzle column builders change (sqlite-core → pg-core)
- Booleans: SQLite used `integer({ mode: 'boolean' })`; pg-core uses native `boolean()`
- `accessibleWeight` stored as `real` — note: Infinity is not used (already stored as `1e10`)
- Drizzle generates new PostgreSQL-compatible migration SQL from the updated schema

### Environment configuration
- `DATABASE_URL` is the single connection string env var
- Local dev: set by `docker-compose.yml` environment block
- Production (Neon): set in Railway/Render/Neon dashboard env vars
- `.env.example` documents `DATABASE_URL=postgresql://user:pass@host:5432/dbname`

### Claude's Discretion
- Connection pool size configuration (postgres.js defaults are reasonable)
- Docker image version for local dev postgres service
- Exact drizzle config file (`drizzle.config.ts`) updates

</decisions>

<specifics>
## Specific Ideas

- The `migrate()` call at server startup stays — same pattern, just with the new driver
- `seedIfEmpty()` should remain in `src/server/db/seed.ts` — only the DB client reference changes
- `docker-compose.yml` should be created at project root for `npm run dev` convenience
- New migration SQL will be generated by `drizzle-kit generate` after schema file is updated

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `src/server/db/schema.ts`: Schema stays structurally identical — column builders swap from `sqliteTable`/`integer`/`real`/`text` to `pgTable`/`integer`/`doublePrecision` or `real`/`text`/`boolean`
- `src/server/db/seed.ts`: `seedIfEmpty()` logic stays the same — only `db` import changes
- `src/server/db/client.ts`: Entire file is replaced — `Database`/`drizzle` from better-sqlite3 → `postgres`/`drizzle` from postgres-js
- `src/server/index.ts`: `migrate()` import path changes; rest of startup sequence unchanged
- `drizzle/0000_brief_madame_hydra.sql`: Existing migration archived — new PostgreSQL migration generated fresh

### Established Patterns
- Drizzle ORM already in use — adapter swap is well-supported by the library
- `migrate()` at startup pattern is preserved; postgres-js migrator is synchronous-compatible
- `.env` file pattern already exists (referenced in `dev:server` script via `--env-file=.env`)

### Integration Points
- `src/server/index.ts` imports `db`, `migrate`, `seedIfEmpty` — all three change import sources
- All API route files import `db` from `./db/client` — those imports stay unchanged (same export name)
- No client-side code touches the database directly — zero frontend changes needed

</code_context>
