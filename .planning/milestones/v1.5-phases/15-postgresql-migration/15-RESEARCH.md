## Summary

Phase 15 is a contained infrastructure swap: replace the SQLite database engine with PostgreSQL without changing the application's public API, business logic, or schema shape. The existing Drizzle ORM layer makes this straightforward — the API surface (`db.select()`, `db.insert()`, `db.transaction()`) is identical across adapters. The work is concentrated in four files: `client.ts` (driver replacement), `schema.ts` (column builder swap), `drizzle.config.ts` (dialect change), and `index.ts` (migrate import path and async startup). The seed and all route files receive only minor adjustments.

The most significant behavioral change introduced by this migration is that **postgres.js is async-by-default**, whereas better-sqlite3 was synchronous. Every database call that currently uses `.all()` or `.run()` (better-sqlite3 synchronous API) must become `await`-able with postgres.js. Drizzle's postgres-js adapter wraps all operations as Promises, so this is purely additive (`async/await`) — no logic changes. The `POST /api/admin/graph` route currently uses `db.$client` to access the raw better-sqlite3 transaction API; this must be replaced with Drizzle's `db.transaction()` which is fully async and natively supported by the postgres-js adapter.

The existing migration SQL (`drizzle/0000_brief_madame_hydra.sql`) is SQLite DDL and must be replaced by a fresh PostgreSQL migration generated by `drizzle-kit generate` after the schema file is updated. The old migration directory content should be cleared before regenerating. The drizzle meta journal (`drizzle/meta/_journal.json`) will also be regenerated. The startup `migrate()` call runs at server boot and applies pending SQL migrations; the postgres-js migrator requires its own dedicated connection (pool size 1) and must `await` the call.

**Primary recommendation:** Replace the four targeted files in this order — schema.ts (type-only change), drizzle.config.ts (dialect), client.ts (driver), index.ts (async startup + migrate path) — then regenerate migrations, add docker-compose, update `.env`, and verify end-to-end with `npm run dev`.

### Pattern 1: postgres.js Client + Drizzle Adapter

**What:** Replace the better-sqlite3 synchronous client with postgres.js async client. Export the `db` handle from `client.ts` identically — all downstream imports are unchanged.

**When to use:** Standard pattern for all Node.js (non-edge) Drizzle + postgres.js setups.

**Example:**
```typescript
// Source: https://rqbv2.drizzle-orm-fe.pages.dev/docs/connect-supabase (verified Context7)
// src/server/db/client.ts  — REPLACES existing file
import { drizzle } from 'drizzle-orm/postgres-js'
import postgres from 'postgres'
import * as schema from './schema'

const client = postgres(process.env.DATABASE_URL!)
export const db = drizzle({ client, schema })
```

Note: The `mkdirSync` call in the current `client.ts` (which ensures the `/data/` SQLite directory exists) is removed entirely — PostgreSQL needs no local file directory.

### Pattern 2: Async Migrate at Startup

**What:** The `migrate()` call moves from synchronous to async. `index.ts` top-level startup code must be wrapped in an async IIFE or top-level `await` (since the file is ESM `"type": "module"`).

**When to use:** Required because postgres-js migrator returns a Promise; better-sqlite3 migrator was synchronous.

**Example:**
```typescript
// Source: https://context7.com/context7/rqbv2_drizzle-orm-fe_pages_dev/llms.txt (verified Context7)
import { drizzle } from 'drizzle-orm/postgres-js'
import { migrate } from 'drizzle-orm/postgres-js/migrator'
import postgres from 'postgres'

// Dedicated connection for migration (max: 1 per Drizzle docs pattern)
const migrationClient = postgres(process.env.DATABASE_URL!, { max: 1 })
const migrationDb = drizzle(migrationClient)

await migrate(migrationDb, { migrationsFolder: './drizzle' })
await migrationClient.end()
```

The production `db` client (exported from `client.ts`) should be a separate pool instance without `max: 1` — the migration client is short-lived and closed immediately after.

### Pattern 3: pg-core Schema Column Builders

**What:** One-for-one swap of sqlite-core builders to pg-core builders. The column names and constraints stay identical.

**When to use:** Every column definition in `schema.ts`.

**SQLite → PostgreSQL builder mapping for this project:**

| SQLite (current) | pg-core (new) | Notes |
|------------------|---------------|-------|
| `sqliteTable` | `pgTable` | Import from `drizzle-orm/pg-core` |
| `text('id').primaryKey()` | `text('id').primaryKey()` | Unchanged |
| `real('x').notNull()` | `real('x').notNull()` | `real` exists in pg-core |
| `integer('floor').notNull()` | `integer('floor').notNull()` | Unchanged |
| `integer('searchable', { mode: 'boolean' }).notNull()` | `boolean('searchable').notNull()` | Native boolean in PostgreSQL |
| `integer('accessible', { mode: 'boolean' }).notNull()` | `boolean('accessible').notNull()` | Native boolean |
| `integer('bidirectional', { mode: 'boolean' }).notNull()` | `boolean('bidirectional').notNull()` | Native boolean |
| `integer('id').primaryKey({ autoIncrement: true })` | `serial('id').primaryKey()` | PostgreSQL uses `serial` for auto-increment |

**Example (verified against Context7):**
```typescript
// Source: https://rqbv2.drizzle-orm-fe.pages.dev/docs/column-types/pg
import { boolean, integer, pgTable, real, serial, text } from 'drizzle-orm/pg-core'

export const nodes = pgTable('nodes', {
  id: text('id').primaryKey(),
  x: real('x').notNull(),
  y: real('y').notNull(),
  label: text('label').notNull(),
  type: text('type').notNull(),
  searchable: boolean('searchable').notNull(),
  floor: integer('floor').notNull(),
  roomNumber: text('room_number'),
  description: text('description'),
  buildingName: text('building_name'),
  accessibilityNotes: text('accessibility_notes'),
})

export const edges = pgTable('edges', {
  id: text('id').primaryKey(),
  sourceId: text('source_id').notNull(),
  targetId: text('target_id').notNull(),
  standardWeight: real('standard_weight').notNull(),
  accessibleWeight: real('accessible_weight').notNull(),
  accessible: boolean('accessible').notNull(),
  bidirectional: boolean('bidirectional').notNull(),
  accessibilityNotes: text('accessibility_notes'),
})

export const graphMetadata = pgTable('graph_metadata', {
  id: serial('id').primaryKey(),          // serial replaces integer autoIncrement
  buildingName: text('building_name').notNull(),
  floor: integer('floor').notNull(),
  lastUpdated: text('last_updated').notNull(),
})
```

### Pattern 4: Drizzle `db.transaction()` Replaces `db.$client.transaction()`

**What:** The current `POST /api/admin/graph` route uses `db.$client` to obtain the raw better-sqlite3 connection for a synchronous transaction. With postgres-js, `db.$client` exposes the postgres.js connection which does not have the same `.transaction()` wrapper. Instead, use Drizzle's own `db.transaction()` which is fully async and supported across all adapters.

**When to use:** Any route that performs multi-step write operations that must be atomic.

**Example (verified Context7):**
```typescript
// Source: https://rqbv2.drizzle-orm-fe.pages.dev/docs/transactions
await db.transaction(async (tx) => {
  await tx.delete(graphMetadata)
  await tx.delete(edges)
  await tx.delete(nodes)
  for (const n of graph.nodes) {
    await tx.insert(nodes).values({ ... })
  }
  for (const e of graph.edges) {
    await tx.insert(edges).values({ ... })
  }
  if (graph.metadata) {
    await tx.insert(graphMetadata).values({ ... })
  }
})
```

The outer `app.post('/api/admin/graph', async (c) => { ... })` handler is already `async` — no change needed to the route registration signature.

### Pattern 5: drizzle.config.ts for PostgreSQL

**What:** Change dialect from `'sqlite'` to `'postgresql'` and point `dbCredentials.url` at `DATABASE_URL` env var.

**Example (verified Context7):**
```typescript
// Source: https://rqbv2.drizzle-orm-fe.pages.dev/docs/drizzle-kit-migrate
import { defineConfig } from 'drizzle-kit'

  },
})
```

### Pattern 7: Async seed.ts

**What:** `seedIfEmpty()` must become `async` because postgres.js queries return Promises. The `.all()` and `.run()` calls from better-sqlite3 do not exist on the postgres-js adapter; instead, Drizzle's postgres-js queries are awaited directly.

**Example (showing changed call pattern):**
```typescript
// Before (better-sqlite3 synchronous):
const existing = db.select().from(nodes).all()

// After (postgres-js async):
const existing = await db.select().from(nodes)

// Before:
db.insert(nodes).values([...]).onConflictDoNothing().run()

// After:
await db.insert(nodes).values([...]).onConflictDoNothing()
```

The `seedIfEmpty()` function signature changes from `function seedIfEmpty(): void` to `async function seedIfEmpty(): Promise<void>`. The call site in `index.ts` gains `await`.

### Anti-Patterns to Avoid

- **Keeping `db.$client.transaction()`:** The better-sqlite3 raw transaction API is gone. Use `db.transaction()` from Drizzle — verified supported in postgres-js adapter.
- **Forgetting `max: 1` on migration client:** Drizzle docs show the migration client with `max: 1`. Using the pooled production client for migrations can cause issues.
- **Not closing the migration client:** Call `await migrationClient.end()` after `migrate()` completes, or it will hold a connection open.
- **Using `.run()` or `.all()`:** These are better-sqlite3 synchronous APIs not present on postgres-js Drizzle queries. All queries must be awaited.
- **Committing the existing `drizzle/` SQL to be reused:** The SQLite DDL (`INTEGER`, `AUTOINCREMENT`, backtick identifiers) is invalid PostgreSQL syntax. The migration folder must be cleared and regenerated.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Connection pooling | Manual pool management | postgres.js built-in pool (`max` option) | postgres.js manages connection lifecycle automatically |
| Migration application | Custom SQL runner at boot | `drizzle-orm/postgres-js/migrator` `migrate()` | Handles idempotency, ordering, journal tracking |
| Transaction isolation | Nested try/catch with manual BEGIN/COMMIT | `db.transaction(async tx => ...)` | Drizzle handles COMMIT/ROLLBACK automatically |
| Schema introspection | Manual column type mapping | `drizzle-kit generate` | Generates correct PostgreSQL DDL from schema.ts |

**Key insight:** Drizzle's adapter abstraction means almost all application code is identical — only the import paths, one or two column builder names, and async/await additions are required.

## Common Pitfalls

### Pitfall 1: Synchronous API Calls After Migration
**What goes wrong:** `db.select().from(nodes).all()` throws at runtime — `.all()` does not exist on postgres-js queries.
**Why it happens:** better-sqlite3 exposed synchronous `.all()` / `.run()` / `.get()` methods. postgres-js queries are Promises; Drizzle's postgres-js adapter returns the query result directly when awaited.
**How to avoid:** Search the codebase for `.all()` and `.run()` calls on `db.*` queries and convert them all to `await db.*`.
**Warning signs:** TypeScript will not catch this at compile time if the return type is `any`. The error only surfaces at runtime: `TypeError: db.select(...).from(...).all is not a function`.

### Pitfall 2: `db.$client` Usage for Transactions
**What goes wrong:** `const sqlite = db.$client; sqlite.transaction(...)` will not work — postgres.js has a different `$client` shape.
**Why it happens:** The current code in `index.ts` (`POST /api/admin/graph`) uses `db.$client` to access the raw better-sqlite3 `Database` instance for its synchronous `.transaction()` method. postgres.js does not have this same API.
**How to avoid:** Replace `db.$client.transaction(...)` with `await db.transaction(async (tx) => { ... })`. The Drizzle `db.transaction()` API is adapter-agnostic and verified working with postgres-js.
**Warning signs:** TypeScript error on `db.$client.transaction` or runtime error if type-cast around it.

### Pitfall 3: SSL Configuration for Neon
**What goes wrong:** Connection to Neon fails with SSL error or times out if `ssl` is not configured.
**Why it happens:** Neon requires SSL. postgres.js defaults `ssl: false` for local connections. The `DATABASE_URL` from Neon includes `sslmode=require` in the connection string, but postgres.js may need the `ssl` option set explicitly to `'require'` to accept cloud certificates.
**How to avoid:** Pass `{ ssl: 'require' }` when in production, or simply include `?sslmode=require` in the connection string and rely on postgres.js parsing it. The most robust pattern is:
```typescript
const client = postgres(process.env.DATABASE_URL!, {
  ssl: process.env.NODE_ENV === 'production' ? 'require' : false,
})
```
**Warning signs:** `Error: self signed certificate` or `ECONNREFUSED` when connecting to Neon.

### Pitfall 4: Old Migration SQL Left in Place
**What goes wrong:** `drizzle-kit generate` detects the old SQLite journal and produces confusing or erroneous output. Or `migrate()` at startup tries to apply SQLite DDL against PostgreSQL and throws a syntax error.
**Why it happens:** The `drizzle/meta/_journal.json` records dialect as `"sqlite"`. drizzle-kit reads this journal to determine what has already been applied.
**How to avoid:** Delete or clear the `drizzle/` folder contents before running `drizzle-kit generate` with the new PostgreSQL config. The new `generate` will create a fresh `0000_*.sql` with valid PostgreSQL DDL.
**Warning signs:** `drizzle-kit generate` completes instantly without generating new SQL, or `migrate()` at startup throws `syntax error at or near "AUTOINCREMENT"`.

### Pitfall 5: `graphMetadata` autoIncrement → serial Mismatch
**What goes wrong:** If `integer('id').primaryKey({ autoIncrement: true })` is left in the pg-core schema, drizzle-kit will generate invalid PostgreSQL DDL.
**Why it happens:** `autoIncrement` is a SQLite-specific option. pg-core uses `serial()` for auto-incrementing integer primary keys.
**How to avoid:** Change to `serial('id').primaryKey()` in schema.ts. This is a **required** change.
**Warning signs:** drizzle-kit generate produces `id integer PRIMARY KEY AUTOINCREMENT` which is invalid PostgreSQL syntax; Drizzle may silently ignore the option or throw.

### Pitfall 6: `migrate()` Not Awaited
**What goes wrong:** The server starts handling requests before migrations are applied, causing `relation "nodes" does not exist` errors.
**Why it happens:** With better-sqlite3 the migrate() call was synchronous. With postgres-js it returns a Promise. If the `await` is missing, the startup sequence continues before the migration completes.
**How to avoid:** Wrap the startup block in a top-level `await` or async IIFE:
```typescript
// ESM top-level await (valid in Node.js ESM modules)
await migrate(migrationDb, { migrationsFolder: resolve(__dirname, '../../drizzle') })
await migrationClient.end()
await seedIfEmpty()
```
The project is already `"type": "module"` — top-level `await` is supported.

## Open Questions

1. **Neon SSL in local Docker dev**
   - What we know: `ssl: false` works for local Docker PostgreSQL (no SSL); `ssl: 'require'` is needed for Neon production
   - What's unclear: Whether the Neon connection string already encodes `sslmode=require` and postgres.js parses it automatically, or whether an explicit `ssl` option is needed in code
   - Recommendation: Use the conditional pattern `ssl: process.env.NODE_ENV === 'production' ? 'require' : false` to guarantee correctness in both environments; lowest-risk approach

2. **`dev:server` script and DATABASE_URL loading**
   - What we know: Current `dev:server` uses `tsx watch --env-file=.env` which loads `.env` natively into `process.env`
   - What's unclear: Whether `docker-compose.yml` injecting `DATABASE_URL` conflicts with `--env-file=.env` or if they're independent (docker-compose sets container env, but dev:server is run on the host machine with tsx, not inside Docker)
   - Recommendation: The server runs on the host (not inside Docker), so `.env` must contain `DATABASE_URL` pointing to `localhost:5432`. docker-compose just runs the DB, not the server. Document this clearly in `.env.example`.

3. **Connection pool size for Neon free tier**
   - What we know: Neon free tier limits connections; postgres.js default `max` is 10
   - What's unclear: Neon free tier exact connection limit (typically 5-25 depending on plan)
   - Recommendation: Claude's discretion — set `max: 5` as a conservative default for the production client to avoid exhausting Neon free tier limits. Can be raised later.
