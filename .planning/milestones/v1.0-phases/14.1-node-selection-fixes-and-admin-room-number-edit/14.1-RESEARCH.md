## Summary

Phase 14.1 is a focused four-change patch against three files. Every change is a small, well-scoped modification to an existing pattern already present in the codebase ‚Äî there is nothing architecturally novel. The existing code has been fully read and each change can be described as a mechanical delta:

1. **Admin node deselect toggle** (`MapEditorCanvas.tsx` `handleNodeClick`): add a guard ‚Äî if the clicked node is already selected in `select` mode, dispatch `SELECT_NODE id: null` and return.

2. **Admin floor-plan-image deselect** (`FloorPlanImage.tsx` + `MapEditorCanvas.tsx`): `FloorPlanImage` has no `onClick` prop today. Two options exist (see Architecture Patterns). The simpler one ‚Äî add an `onClick` prop to `FloorPlanImage` and attach it to the Konva `<Image>` ‚Äî is safe and matches the file's existing optional-prop pattern.

3. **Student A/B pin tap-to-deselect** (`SelectionMarkerLayer.tsx`): the component is currently display-only ("no click handlers" per its own JSDoc comment). Adding `onClick`/`onTap` to the two pin Groups ‚Äî calling `clearStart` or `clearDestination` from props ‚Äî is a minimal, pattern-consistent change. The `hitFunc` pattern used in `LandmarkMarker` and `NodeMarkerLayer` provides a bigger tap target; the CONTEXT.md already flags this as a discretion item.

4. **NodeDataTable room number inline edit + SearchOverlay secondary label**: `NodeDataTable` already has a `editingCell` state typed as `{ rowId: string; field: 'label' | 'type' }`. Extending the union to `'label' | 'type' | 'roomNumber'` and replicating the `<input>` pattern from the `label` cell is the path of least resistance. `SearchOverlay` already renders `node.roomNumber` in a `<span>` if present (line 307); the ask is to ensure that span is visible and well-styled.

**Primary recommendation:** All four changes are surgical ‚Äî edit three files with a combined estimated delta of ~60 lines. No new dependencies, no new hooks, no schema changes.

### Pattern 1: Admin node toggle deselect

**What:** `handleNodeClick` in `MapEditorCanvas.tsx` currently dispatches `SELECT_NODE id: nodeId` unconditionally in select mode. Adding a toggle:

```typescript
// Source: MapEditorCanvas.tsx handleNodeClick (lines 185-218)
const handleNodeClick = useCallback(
  (nodeId: string) => {
    if (state.mode === 'select') {
      // Toggle deselect: clicking already-selected node clears selection
      if (state.selectedNodeId === nodeId) {
        dispatch({ type: 'SELECT_NODE', id: null })
      } else {
        dispatch({ type: 'SELECT_NODE', id: nodeId })
      }
    }
    // ... rest of add-edge handling unchanged
  },
  [state.mode, state.selectedNodeId, /* existing deps */ dispatch, recordHistory],
)
```

**Side-panel closes automatically** ‚Äî `EditorSidePanel` receives `selectedNode` derived from `state.selectedNodeId`; setting it to `null` makes `selectedNode` null and the panel renders nothing.

**Dependency array note:** `state.selectedNodeId` must be added to the `useCallback` dep array (currently absent because it was not needed).

### Pattern 2: Admin floor-plan-image deselect

**What:** When the user clicks the floor plan image (not the transparent stage background) in select mode, the selection should clear. Currently `handleStageClick` only fires when `e.target === stage` (the transparent stage background), not when the image shape is clicked.

**Option A (recommended): Add `onClick` prop to `FloorPlanImage`**

```typescript
// FloorPlanImage.tsx ‚Äî add optional onClick prop
interface FloorPlanImageProps {
  // ... existing props
  onClick?: () => void
}

// In JSX:
<Image
  ref={imageRef}
  image={image}
  x={rect.x}
  y={rect.y}
  width={rect.width}
  height={rect.height}
  onClick={onClick}
/>
```

```typescript
// MapEditorCanvas.tsx ‚Äî wire it up
<FloorPlanImage
  image={image}
  isFullLoaded={isFullLoaded}
  viewportWidth={width}
  viewportHeight={editorHeight - 52}
  onImageRectChange={setImageRect}
  onClick={() => {
    if (state.mode === 'select') {
      dispatch({ type: 'SELECT_NODE', id: null })
      dispatch({ type: 'SELECT_EDGE', id: null })
    }
  }}
/>
```

This is clean and matches the existing optional-callback pattern (`onImageRectChange` is already optional). The `onClick` fires when the image shape is clicked; it does not conflict with `handleStageClick` because `e.target === stage` is false when an Image shape is clicked.

**Option B (alternative): Check `e.target` in `handleStageClick`**

Add a check in `handleStageClick` for whether `e.target` is a Konva Image node. More fragile because it relies on checking Konva node type internally rather than a clean prop interface.

**Option A is the right choice.**

**Hit area expansion (discretion item):** The `hitFunc` pattern used in `LandmarkMarker` and `NodeMarkerLayer` expands the hit area to `radius * 2.5`. The SelectionMarkerLayer A/B circles use `PIN_RADIUS = 12`. Counter-scaling means the visual size is constant at 12px screen-pixels. A `hitFunc` on the `<Circle>` expanding to `PIN_RADIUS * 2.5 = 30` screen-pixels is advisable for touch UX:

```typescript
  fill={START_COLOR}
  stroke="#ffffff"
  strokeWidth={2}
  hitFunc={(context, shape) => {
    context.beginPath()
    context.arc(0, 0, PIN_RADIUS * 2.5, 0, Math.PI * 2, true)
    context.closePath()
    context.fillStrokeShape(shape)
  }}
/>
```

`clearStart()` already sets `activeField` to `'start'` (line 82-84 in `useRouteSelection.ts`). `clearDestination()` sets `activeField` to `'destination'` (line 86-89). This satisfies the CONTEXT.md requirement exactly ‚Äî no changes needed to `useRouteSelection`.

### Pattern 4: NodeDataTable room number inline edit

**Current state:** `editingCell` is typed as `{ rowId: string; field: 'label' | 'type' } | null`. The Room # cell (line 187) renders `node.roomNumber ?? '‚Äî'` as static text.

**Change 1: Extend editingCell union type**

```typescript
// NodeDataTable.tsx line 36
const [editingCell, setEditingCell] = useState<{
  rowId: string
  field: 'label' | 'type' | 'roomNumber'
} | null>(null)
```

**Change 2: Extend commitEdit to handle roomNumber**

```typescript
function commitEdit(node: NavNode, field: 'label' | 'type' | 'roomNumber') {
  if (field === 'label') onUpdateNode(node.id, { label: editValue })
  else if (field === 'type') onUpdateNode(node.id, { type: editValue as NavNodeType })
  else if (field === 'roomNumber') {
    // Empty string ‚Üí null (no room number)
    onUpdateNode(node.id, { roomNumber: editValue.trim() === '' ? undefined : editValue.trim() })
  }
  setEditingCell(null)
}
```

Note: `NavNode.roomNumber` is typed as `string | undefined` (optional field in `NavNodeData`). Setting to `undefined` removes it. The CONTEXT.md says "empty string treated as null" ‚Äî `undefined` is the correct TypeScript representation for an absent optional field.

**Change 3: Replace static Room # cell with inline-edit cell (mirrors label cell pattern)**

```typescript
{/* Room # ‚Äî inline editable */}
{/* biome-ignore lint/a11y/useKeyWithClickEvents: td stopPropagation only, actual interactive elements inside handle keyboard */}
<td className="px-3 py-2" onClick={(e) => e.stopPropagation()}>
  {editingCell?.rowId === node.id && editingCell.field === 'roomNumber' ? (
    <input
      // biome-ignore lint/a11y/noAutofocus: intentional focus for inline edit UX
      autoFocus
      value={editValue}
      onChange={(e) => setEditValue(e.target.value)}
      onBlur={() => commitEdit(node, 'roomNumber')}
      onKeyDown={(e) => {
        if (e.key === 'Enter') commitEdit(node, 'roomNumber')
        if (e.key === 'Escape') setEditingCell(null)
      }}
      className="w-full rounded border border-blue-400 px-1 py-0.5 text-sm focus:outline-none"
    />
  ) : (
    // biome-ignore lint/a11y/useKeyWithClickEvents: inline edit trigger; keyboard handled by surrounding input on activation
    // biome-ignore lint/a11y/noStaticElementInteractions: span is an inline edit trigger
    <span
      className="block cursor-text rounded px-1 py-0.5 text-gray-500 hover:bg-blue-100"
      onClick={() => {
        setEditingCell({ rowId: node.id, field: 'roomNumber' })
        setEditValue(node.roomNumber ?? '')
      }}
    >
      {node.roomNumber ?? '‚Äî'}
    </span>
  )}
</td>
```

The existing `commitEdit` for `label` does not handle Escape ‚Äî it only commits on Enter and blur. CONTEXT.md says "Escape cancels (or blur commits ‚Äî match existing pattern)". The existing `type` field (select element) relies solely on blur. For `roomNumber` (a text input like `label`), adding Escape-cancel is the better UX and not prohibited. The label cell can optionally get the same treatment for consistency, but the minimum is to match the label cell exactly and optionally add Escape on top.

### Anti-Patterns to Avoid

- **Do not add `state.selectedNodeId` to `handleNodeClick`'s closure without updating the dep array.** The current `handleNodeClick` dep array does not include `state.selectedNodeId`. The toggle check reads `state.selectedNodeId`, so it must be added to `useCallback` deps or the stale closure will not see the current selection.

- **Do not mutate the Konva stage directly for deselect.** All selection state flows through `dispatch({ type: 'SELECT_NODE', id: null })` via the `useEditorState` reducer. Don't bypass the reducer.

- **Do not set `roomNumber: ''` on commit.** The NavNode type has `roomNumber?: string`. An empty string is semantically different from absent. Setting `undefined` (not empty string) removes the field from the object. The CONTEXT.md explicitly says "empty string treated as null."

- **Do not add `onClick` to the Konva Stage for floor-plan-image deselect.** The Stage already has `onClick={handleStageClick}` which only fires for stage-target clicks. Adding a second Stage-level onClick would duplicate logic. The clean path is the `FloorPlanImage` prop.

## Common Pitfalls

### Pitfall 1: Stale closure in handleNodeClick

**What goes wrong:** `handleNodeClick` is a `useCallback` with a dep array. Currently `state.selectedNodeId` is NOT in its dep array because it was not read inside the callback. Adding the toggle check reads `state.selectedNodeId`. If the dep array is not updated, the closure captures the value at creation time and the toggle comparison is always stale.

**How to avoid:** Add `state.selectedNodeId` to the `useCallback` dep array for `handleNodeClick`.

**Warning signs:** Toggle doesn't work on second click, or always deselects regardless of what was previously selected.

---

### Pitfall 2: FloorPlanImage onClick fires in add-node/add-edge modes

**What goes wrong:** The `FloorPlanImage` `onClick` would be wired to clear selection. But in `add-node` mode, clicking the image should place a node (handled in `handleStageClick`). In `add-edge` mode it should start/complete an edge. If the onClick unconditionally clears, it interferes.

**How to avoid:** Guard the `FloorPlanImage` onClick callback with `if (state.mode === 'select')` ‚Äî only deselect in select mode.

```typescript
onClick={() => {
  if (state.mode === 'select') {
    dispatch({ type: 'SELECT_NODE', id: null })
    dispatch({ type: 'SELECT_EDGE', id: null })
  }
}}
```

In `add-node` mode, `handleStageClick` handles placement because the stage's onClick fires on any click ‚Äî including clicking the image shape (the image click also bubbles up to the stage). The `FloorPlanImage` onClick would fire first, then the stage onClick. In `add-node` mode the `FloorPlanImage` guard prevents interference; the stage onClick then places the node.

**Warning signs:** Nodes cannot be placed by clicking the floor plan image; clicking image in add-edge mode cancels pending edge.

---

### Pitfall 3: click event bubbling from FloorPlanImage to Stage in select mode

**What goes wrong:** When the `FloorPlanImage` Image shape is clicked, the event bubbles to the Stage's `onClick` (`handleStageClick`). In `handleStageClick`, the guard is `if (e.target === stage)` ‚Äî which is false for Image clicks ‚Äî so the stage handler does nothing. The `FloorPlanImage` onClick fires and deselects. This is the correct intended behavior. No double-deselect issue exists.

**Warning signs:** None expected, but if deselect fires twice (causing re-select), add `e.cancelBubble = true` in the FloorPlanImage onClick handler.

---

### Pitfall 4: A/B pin click bubbles to LandmarkLayer/Stage

**What goes wrong:** The A/B pins in `SelectionMarkerLayer` sit above `LandmarkLayer`. If the pin is clicked and the event bubbles, it might trigger a landmark tap (which would call `setFromTap` and re-assign the cleared slot).

**How to avoid:** The A/B pins replace the landmark markers for selected nodes (`hiddenNodeIds` filtering in `LandmarkLayer`). So if pin A covers node X, node X's `LandmarkMarker` is not rendered. However, to be safe, call `e.cancelBubble = true` in the Group onClick for the A/B pins ‚Äî matching the pattern used in `NodeMarkerLayer` (line 67-68).

---

### Pitfall 5: roomNumber undefined vs empty string in onUpdateNode

**What goes wrong:** `onUpdateNode` is typed as `(id: string, updates: Partial<NavNode>) => void`. Passing `{ roomNumber: undefined }` may or may not remove the field depending on how the reducer handles it.

**How to avoid:** Check `useEditorState`'s `UPDATE_NODE` reducer to confirm it merges with spread. It does: `{ ...node, ...changes }` ‚Äî spreading `{ roomNumber: undefined }` over a node that has `roomNumber: '204'` will set it to `undefined`, effectively removing it. TypeScript strict null checks: `Partial<NavNode>` includes `roomNumber?: string` which accepts `undefined`.

## Open Questions

1. **Should Escape cancel inline edit for label/type cells too (for consistency)?**
   - What we know: The existing label cell has no Escape handler; it only commits on blur/Enter.
   - What's unclear: CONTEXT.md says "Escape cancels (or blur commits ‚Äî match existing pattern)" ‚Äî this means either behavior is acceptable.
   - Recommendation: Add Escape-cancel to the roomNumber cell only. Modifying label/type behavior is out of scope for this phase.

2. **Does UPDATE_NODE spread handle `roomNumber: undefined` correctly?**
   - What we know: The reducer uses `{ ...node, ...changes }` pattern (standard for this codebase per Phase 10 decisions). Spreading `{ roomNumber: undefined }` onto an object with `roomNumber: '204'` in JavaScript sets the key to `undefined` ‚Äî it does NOT delete the key. This means the field would be present as `undefined` in the object.
   - What's unclear: Whether the API serialization (JSON.stringify) omits `undefined` values. JSON.stringify does omit `undefined` values ‚Äî so the field would be absent in the JSON sent to the server.
   - Recommendation: This is fine for the client-side state and API save. No changes needed to the reducer.
